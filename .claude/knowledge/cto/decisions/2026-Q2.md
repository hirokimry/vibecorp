## 2026-05-09 — PR #534 — workflow 削除 PR 自身を実機検証として扱う設計判断（GitHub Actions ランタイムアサーション却下）

### 判断内容

GitHub Actions ワークフローファイルを削除する PR（#534）において、「ワークフローが起動しないこと」の検証方法について判断した。

### 結論: 静的保証 + statusCheckRollup 証跡を「実機検証」として採用。ランタイムアサーションは却下

GitHub Actions の `pull_request` トリガーは PR HEAD ブランチの workflow ファイルを参照して起動を解決する。PR HEAD でファイルが存在しない場合、トリガー解決対象が無く起動不能となる（GitHub Actions 仕様）。この静的保証に加え、`statusCheckRollup` に AI Review 系ジョブが存在しない事実が実機証跡となる。

### 却下した代替案

GitHub Actions API を用いた workflow runs のランタイムアサーション（`assert: AI Review ジョブが起動していない`）は以下の理由で却下:

- 権限・タイミング・リトライ設計を伴う heavy lift
- 削除 PR という性質上、静的保証が既に十分（ファイル不在 = 起動不能は仕様から導出可能）
- Issue #532 の `intent/feature` スコープを超える工数

### 適用ルール

ワークフローファイルを削除する PR で「起動しないことを証明する」テストを書く必要はない。PR HEAD にファイルが存在しないことと `statusCheckRollup` の証跡で十分。

## 2026-05-08 — PR #520 — race condition 修正における単一 PR 方針の堅持（分割却下）

### 判断内容

PR #520（CI race condition 解消）において、「ship → pr への責務委譲」と「`pr-intent-inherit.yml` CI 削除」を別 PR に分割するよう指摘された。

### 結論: 分割却下。一括対応が正しい設計

2 つの変更は同一の bug 修正を達成するための**分離不可能な構成要素**であり、分割すると以下の中間状態が発生する:

- 分割順が「委譲先行 → CI 削除」の場合: CI は生き残ったまま、ship も委譲済みで二重処理が発生
- 分割順が「CI 削除先行 → 委譲後」の場合: 全 PR が無ラベルで fail する状態が発生

いずれも main の安定性を損なうため、`intent/bugfix` の「最小修正で直す」重視軸として一括対応が適切。`intent/refactor` / `intent/infra` への再分類は不要。

### 教訓

bug の根本原因（責務逆転）と保険的経路（CI 肩代わり）が密結合している場合、分割による中間状態リスクが一括対応コストを上回ることがある。分割の指摘を受けた際は中間状態の安全性を先に評価する。


## 2026-05-07 — PR #507 — CI ワークフローの fail-fast 設計（graceful skip 却下）

### 判断内容

`templates/.github/workflows/ai-review.yml` の `claude-review` ジョブで、`secrets.CLAUDE_CODE_OAUTH_TOKEN` 未登録時に graceful skip（ジョブを `if` で条件スキップして PR を緑にする）する案を却下した。

### 結論: fail-fast 設計を維持

`claude_action.enabled: true` にしている利用者は `CLAUDE_CODE_OAUTH_TOKEN` を登録する責任を持つ。graceful skip を採用すると:

- トークン未登録でも PR が緑になる
- claude-action が「動いていない」ことに気づけない
- 「緑なのに機能が動作していない」状態が生まれる

fail-fast（トークン未登録 = PR 赤）の方が利用者に問題を確実に伝えられる。登録忘れの早期発見としても機能する。

### 適用パターン

外部シークレットや認証情報に依存する機能を有効化する場合、**preflight guard による graceful skip ではなく fail-fast を基本とする**。「エラーを隠す」設計は利用者の問題発見を遅らせる。

  3. macOS の `sandbox-exec` / Linux の `bwrap` はカーネル機能のため起動オーバヘッド極小（数十 ms 以下）で、bind mount 不要
  4. Issue #279（コンテナ実行のクイックスタート簡素化）が解決しようとしていた「コンテナ実行の複雑さ」は、コンテナ自体を廃止することで根本的に消滅する

  - PR #278 (`feature/container-isolation`) は close 済み・unmerged のまま放置
  - `docker/` ディレクトリ・`install.sh` の docker 参照は撤去済み（既に main にマージ済み）

## 2026-05-06 — Epic #455 — claude-code-action CHANGES_REQUESTED 残存問題の構造的解決方針

### 真因

claude-code-action は GitHub の「review submission」モデルを使う（`gh pr review --request-changes` / `--approve`）。GitHub の仕様上、`CHANGES_REQUESTED` を発行したレビュアーが明示的に `APPROVE` または `DISMISS` を発行しない限り、PR の `reviewDecision` は `CHANGES_REQUESTED` のまま残存する。

CodeRabbit はこれをサーバーサイド状態管理で解決している。push 毎に前回の `CHANGES_REQUESTED` を自動 dismiss または re-approve する。claude-code-action にはそのサーバーサイド状態管理がない。

REVIEW.md の現行 Step 7 は「指摘件数 ≥ 1 → CHANGES_REQUESTED、0 件 → approve」と定義しているが、「前回発行した CHANGES_REQUESTED をクリアする責務」が抜けている。全スレッドが resolved になっても、CHANGES_REQUESTED レビュー自体は別エンティティとして残存するため。

### CodeRabbit との構造的差異

| 観点 | CodeRabbit | claude-code-action |
|------|-----------|-------------------|
| review state 管理 | サーバーサイドで前回レビューを dismiss してから再評価 | プロンプト指示のみ、前回 review state を自律管理しない |
| スレッド resolved 後の挙動 | 自動で approve に遷移 | CHANGES_REQUESTED を維持したまま次レビューを出す |

### 解決策（優先度順）

**案A（採用）: REVIEW.md に「前回 CHANGES_REQUESTED → 全スレッド resolved なら approve 発行」を明示する**

REVIEW.md の Step 7 の前に「前回レビュー状態確認」ステップを追加し、全 review thread が resolved かつ修正対象 0 件の場合は明示的に `gh pr review --approve` を発行する指示を入れる。「新しい push 後は前回の CHANGES_REQUESTED を上書きして再評価する」という指示を明文化する。

**案B（次善）: ワークフロー側で gh api を使い claude の前回レビューを事前 dismiss する**

push 毎に `gh api .../reviews/{id}/dismissals` で claude の前回レビューを dismiss してから claude-code-action を起動する。技術的には可能だが、dismiss が先走ると「一瞬 approve 状態になる」タイムラグが発生する。

**案C（却下）: claude-code-action@v1 の dismiss-stale-reviews 相当オプションを探す**

claude-code-action@v1 の公式 API に該当パラメータは存在しない。実装不可能。

### 採用根拠（案A）

変更箇所は REVIEW.md と templates/REVIEW.md.tpl のみ（ワークフロー変更不要）。claude-code-action の公式 API 外の操作を一切使わない。ただし LLM がプロンプト指示を確実に実行するかはプロンプト品質依存であり、GitHub サーバーサイド制御と異なる保証強度であることを認識する。100% の構造保証が必要なら案Bを案Aに重ねる二重防御も可能。

### エスカレーション事項

「プロンプト指示で確実に動くか」は LLM の解釈次第であり、CodeRabbit のサーバーサイド実装とは保証強度が根本的に異なる。技術スタックの制約として CEO への説明が必要。

## 2026-05-03 — vibecorp 独自 severity 5段階の厳密定義設計

### 対象

`review-criteria.md` が参照する CodeRabbit 公式 severity の曖昧さを排し、vibecorp 文脈で客観的判定可能な厳密定義を設計する（OSS プラグイン + AI エージェント基盤の文脈）。

### 背景・課題

現行の `review-criteria.md` は「どのカテゴリに何が入るか」は書いてあるが境界が曖昧。特に以下の判定で迷いやすい：
- フックのエラーハンドリング欠落は Critical か Major か
- `.claude/rules/` 違反は Minor か Trivial か
- `set -e` 未使用は Major か Trivial か

### 設計判断

各 severity の境界線を以下の客観条件で定義する：

**Critical（マージ不可・即時修正）**
1. フック・スキルがセキュリティ保護を無効化・迂回できる
2. データ破壊（`.claude/knowledge/` 直接消去等）を引き起こせる
3. `ANTHROPIC_API_KEY` / 認証情報が平文でコードに混入している
4. ガードレール（`protect-files.sh` 等）が機能しなくなる変更

**Major（マージ不可・修正必須）**
1. バグ：特定の入力でフック・スキルが意図しない終了コードを返す
2. `set -euo pipefail` 下で unbound variable になりうる
3. リソースリーク（一時ファイル / ワークツリーが cleanup されない）
4. 既存テストが壊れる・新機能にテストがない（`testing.md` 違反）

**Minor（マージ前に対応推奨）**
1. `.claude/rules/` 規約違反（`shell.md`・`comments.md` 等）
2. `docs/` 仕様書の MUST/MUST NOT 違反
3. エラーメッセージが英語（`workflow.md` 日本語統一違反）
4. ハードコードされた値で環境依存が生じる
5. コメントと挙動の不一致

**Trivial（マージ可・対応推奨）**
1. スタイル統一（インデント・空白・クォート）
2. 命名の改善提案（機能は正しいが名前が冗長）
3. 軽微な最適化（実害なし）

**Info（対応不要）**
1. 情報提供のみ（判断・変更を求めない）
2. 変更範囲外の既存コードへの指摘
3. 誤検知・文脈を理解していない指摘

### 境界判定の軸

- Critical vs Major: 「悪意ある/偶発的操作で他ユーザー環境に影響が出るか」
- Major vs Minor: 「今この PR でユーザーが損害を受ける入力パスが存在するか」
- Minor vs Trivial: 「`.claude/rules/` または `docs/` の MUST 条件に違反しているか」
- Trivial vs Info: 「対応するとコードが改善されるか」

### 判定

定義は設計済み。`review-criteria.md` の更新（管轄ファイル `.claude/rules/` 配下）として反映することを推奨。

## 2026-05-02 — CodeRabbit 代替評価（A: claude-code-action / B: Qodo PR-Agent OSS / C: カスタム強化）

### 評価背景

CEO 要求: CodeRabbit ($20/月) の代替を「ローコスト・あわよくば Claude Max 定額内・Claude クレジットも使わないなら最高」という優先順位で技術評価。

### 4契約の再現性評価（`/vibecorp:pr-fix-loop` との整合）

| 契約 | A (claude-code-action+OAuth) | B (Qodo PR-Agent OSS, AGPL-3.0) | C (カスタム強化) |
|------|------------------------------|----------------------------------|-----------------|
| auto-review（push 時自動起動） | ◎ GitHub Actions trigger | ◎ `pull_request: synchronize` trigger | △ GitHub Actions 追加が必要 |
| approve/request changes 遷移 | △ 自前 `gh pr review` 必要、公式未保証 | △ 機能フラグあり、APPROVE/REQUEST_CHANGES 出力は未確認 | △ 同左（自前実装必須） |
| auto-resolve（修正済みコメント resolve） | × 他者コメントの resolve は API 制約あり | △ 自己コメントのみ | △ `gh api` dismiss で代替可 |
| 日本語レビュー出力 | ◎ system prompt で制御 | ◎ LLM prompt で制御 | ◎ カスタムレビュアーは日本語前提 |

**共通制約**: `GITHUB_TOKEN` は自 PR へのレビューステータス発行が禁止。A・B・C いずれも Bot 用 PAT または GitHub App token が必要。

### 判断

**第一推奨: C（カスタム強化）**

- **A（OAuth）**: 公式が「重い自動化は対話用クォータを枯渇」「チーム自動化は API Key 推奨」と明示警告。vibecorp の自動化スケールでは不向き
- **B（PR-Agent AGPL-3.0）**: vibecorp が OSS 公開前提のため、ラップ・改変コードへのライセンス感染リスク。CLO 判断なしに採用不可
- **C**: コストゼロ・ライセンスリスクなし。`/vibecorp:pr-fix-loop` は `reviewDecision` フィールドを polling するだけで動く設計のため、レビュアーが誰でも接続コスト最小

### 設計方針（技術的負債の記録）

`coderabbit.enabled: bool` の現設計はプロバイダが増えると拡張困難。将来 `vibecorp.yml` に以下を追加する方向が自然:

```yaml
review_provider:
  type: coderabbit | claude-action | self-hosted | none
```

`coderabbit.enabled: bool` → `review_provider.type: string` への移行を技術的負債として認識。Issue 起票を推奨。

### 各検討事項への技術判断

#### 1. 整合性評価（構成 A vs B の違いは正当か）

**判断: 部分的に正当だが、設計意図が未文書化のため技術的負債**

分析員（accounting / security）は C*O とは異なり「判断を蓄積する」のではなく「監査レポートを生産する」という責務差は存在する。この差を構成に反映すること自体は正当。ただし、その判断が明文化されていないため、新しいエージェントが追加された際に一貫した判断ができない。

#### 2. 件数増加時の挙動（5 年 300 件）

**判断: フラット構成は長期運用に耐えない**

`security/audit-log.md` が現在採用している single-file append-only 方式は、Issue #335 で構成 A の移行を決断した理由（631 行 / 約 40KB でトークンコストが問題化）と同一の問題に収束する。監査が本格稼働した場合、年間 60 件（週次 52 件 + 月次 12 件）で 5 年後 300 件のデータが単一ファイルに詰まる。

`accounting/audit-YYYY-MM-DD.md` 方式（問題文の想定）でも、ディレクトリフラットに 300 ファイルが並ぶ構造は ls や一覧性の観点で劣化する。

#### 3. 責務領域（機能領域別 vs 役割別の混在）

**判断: 現状の混在は許容できるが、整理の方向性を定める必要あり**

`{role}/` は「判断の主体（エージェント）」単位であり、`accounting/` `security/` は「機能領域（分析員の専門）」単位。これらが同一 `knowledge/` 配下に混在しているが、実際には `accounting = 経理分析員` `security = セキュリティ分析員` であり、役割単位として読み替えると一貫している。混在の問題というより、命名規則が統一されていない問題。

#### 4. cycle-metrics の特殊性（揮発データの永続パス保存）

**判断: 問題あり（ただし現実には未実装のため影響なし）**

揮発データ（サイクルごとにリセットされるメトリクス）を永続 git 履歴に載せるのは本質的に誤り。これは `.claude/knowledge/` ではなく `~/.cache/vibecorp/state/<repo-id>/` に保存すべきデータ性質。問題文が記載する `cycle-metrics-YYYY-MM-DD.md` は現実には存在しないファイルだが、将来実装する際は設計判断が必要。

#### 5. あるべき構成（提案）

**推奨案: 責務別 2 段構成に統一 + テンプレート分離**

```text
.claude/knowledge/
├── {c-suite-role}/          ← C*O ロール（cto / cpo / cfo / ciso / clo / sm）
│   ├── decisions-index.md   ← 毎起動 Read、小さく保つ
│   ├── decisions/
│   │   └── YYYY-QN.md
│   └── {role}-principles.md ← 静的原則定義（読取専用）
│
├── {analyst-role}/          ← 分析員ロール（accounting / security / legal）
│   ├── {role}-principles.md ← 静的原則定義
│   ├── {role}-audit-template.md ← 雛形（静的リソース）
│   └── audit-log/           ← 監査ログ専用ディレクトリ
│       └── YYYY-QN.md       ← 四半期単位で集約
│
└── {shared}/                ← ロール横断の共有知識（現状のトップレベル .md）
    ├── agents-vs-skills.md
    ├── mvp-vs-preset.md
    └── ...
```

**設計根拠:**

- C*O の `decisions/` と分析員の `audit-log/` は同じ「時系列蓄積文書」として統一的に扱える
- 四半期単位の集約は `docs/migration-decisions-index.md` で実証済みのパターンを流用
- 監査ログに index を設けないのは、監査レポートはサマリーが自己完結しており「今回に関連する過去監査」を検索するユースケースが C*O の判断記録より低いため
- `{role}-principles.md` と `audit-log/` を分離することで、起動時 Read コストを原則ファイルのみに限定できる
- `protect-knowledge-direct-writes.sh` の deny パターンは `decisions/` → `audit-log/` へのパス変更が必要

#### 6. 移行コスト

**対象ファイル数（現状）:**

- 変更が必要なファイル:
  - `security/audit-log.md` → `security/audit-log/2026-Q2.md` に分割（1 ファイル）
  - `security/security-audit-template.md` → 命名維持でも可
  - `accounting/cost-audit-template.md` → 命名維持でも可
- `protect-knowledge-direct-writes.sh` のパターン変更が必要: `audit-.+\.md$` → `audit-log/` パス
- テンプレートの更新が必要: `templates/claude/hooks/protect-knowledge-direct-writes.sh`

**現時点での実際の影響:**
- `accounting/audit-YYYY-MM-DD.md` は未実装のため移行対象なし
- `security/audit-log.md` の分割は現在 1 ファイルのため低コスト
- 主なコストはフックのパターン変更とドキュメントの整備

#### 7. buffer 経由統一（Issue #439）との整合性

**判断: パターン変更は必要だが、構成変更と同時に行うべき**

現在のフックパターン:
- `^[^/]+/decisions/` — C*O 判断記録を deny（buffer 必須）
- `^audit-.+\.md$` — 監査レポートを deny（buffer 必須）

`audit-log/YYYY-QN.md` 構成に変更すると `^[^/]+/audit-log/` パターンに変更が必要。構成変更と同時に `protect-knowledge-direct-writes.sh` を更新することで整合を保てる。

### 結論（あるべき構成）

**Phase 1（今すぐ）: ドキュメント整備**

構成 A（C*O ロール）は正しい設計。構成 B（分析員）の設計意図を `docs/design-philosophy.md` に追記し、技術的負債を明文化する。特に以下を記述:
- `{c-suite-role}/` と `{analyst-role}/` の責務差
- 監査ログの四半期集約方針
- cycle-metrics は `~/.cache/` に置くこと

**Phase 2（Issue として起票）: 監査ログの構造化**

`security/audit-log.md` を `security/audit-log/2026-Q2.md` に移行し、`accounting/` も同様の構造を整備。`protect-knowledge-direct-writes.sh` のパターンも同時に更新。

**Phase 3（将来・YAGNI 境界）: 統一 index 検討**

監査ログが 100 件を超えたタイミングで、監査ログにも index 構造を設けるか評価する（現時点では YAGNI）。

### リスク

1. **パターン変更タイミングの不整合**: 構成変更（`audit-log.md` → `audit-log/YYYY-QN.md`）とフック更新が別 PR で行われると、期間中に deny パターンが効かないか過剰に deny するウィンドウが生まれる。同一 PR での変更を強く推奨。

2. **テンプレート配布の更新漏れ**: `protect-knowledge-direct-writes.sh` は `templates/claude/hooks/` 配下のテンプレートとして配布される。本体の hook 更新だけでなく、テンプレート側の更新も必須。`docs/migration-decisions-index.md` 相当のドキュメントを `accounting/` `security/` 向けにも整備する必要がある。

3. **cycle-metrics の実装時の混入**: 将来 cycle-metrics スキルが実装される際に `.claude/knowledge/` に書き込む設計が採用されると、揮発データが git 履歴に乗る問題が発生する。`docs/design-philosophy.md` に永続パス vs キャッシュパスの分類基準を先に記述しておくことでリスクを抑制できる。

## 2026-04-05 — team-auto-approve.sh — Bash コマンドの多段検証とセグメント分割（Issue #233 対応）

## 2026-04-08 — team-auto-approve.sh — quote-aware セグメント分割への置き換え（Issue #252）

- **判断**: `sed 's/&&/\n/g; s/;/\n/g'` による単純分割を廃止し、awk で `in_single`/`in_double` フラグを管理する quote-aware 分割に置き換える
- **根拠**: `awk '/^key:/ { sub(...); print; exit }' file` のような正当なコマンドが `;` の位置で誤分割され、safe list 判定に漏れて permission ダイアログが毎回表示される問題が発生した。quote 内の区切り文字を無視しない実装はバグと同等であり、フックの信頼性を損なう

## 2026-04-16 — vibecorp-sandbox に symlink 解決 + 2 段階検証と WORKTREE ⊇ HOME 拒否を追加（PR #317 第 2 回レビュー対応）

- **判断**: `canonicalize_dir()` ヘルパ（`(cd "$p" && pwd -P)`）を追加し、raw バリデート → canonicalize → canonicalize 後に再度 `validate_abs_path` を実行する 2 段階検証を採用。加えて、canonicalize 後に `case "${HOME_VALUE}/" in "${WORKTREE_VALUE}/"*)` で WORKTREE が HOME を包含するケースを入口で拒否する
- **根拠**:
  - macOS の `$TMPDIR` は `/var/folders/...` で `/private/var/...` の symlink であり、解決前後の文字列を混在比較すると包含判定が崩れる。canonicalize 後に比較することで symlink の違いによる誤判定を防ぐ
  - `WORKTREE=/Users` のような設定は sandbox-exec の `(subpath (param "WORKTREE"))` を通じて `~/.ssh` / `~/.aws` を RW 境界に含めてしまう。書込み境界が HOME 全体に広がるのを防ぐため、入口で拒否する
  - WORKTREE ⊇ HOME 包含判定に bash の `case` 文を使った理由: macOS のデフォルト bash は 3.2 であり `[[ =~ ]]` の正規表現は使えないが、`case` の glob パターンは bash 3.2 でも安定して動作する。`case "${HOME}/" in "${WORKTREE}/"*)` でサフィックス `/` を付けることで完全パスセグメント単位の一致判定ができ、誤マッチを防げる
- **代替案**: `[[ "$HOME_VALUE" == "$WORKTREE_VALUE"/* ]]` のような bash 4+ の二重ブラケット glob 展開も機能するが、macOS 3.2 互換性を確保するため `case` 文を採用した

## 2026-04-16 — claude TUI ハング修正 — sandbox-exec プロファイルへの `file-ioctl` 追加（Issue #320）

## 2026-04-16 — ゲートスタンプ XDG 移行に伴う実装上の技術判断（PR #327）

- **判断**: (a) `basename | tr` パターンには `printf '%s'` で trailing newline を除去する。(b) PreToolUse フックのスタンプパス計算は early exit の後に置く。(c) 共通ヘルパーの重複呼び出しは `STAMP_DIR` 変数で1回取得・再利用に統一。(d) sandbox subpath 外の親ディレクトリは sandbox-exec より前（install.sh / pre-launch）に事前作成する
- **根拠**:
  - `basename "$root" | tr -cs ...` は basename の trailing `\n` も `_` に変換して末尾に付く。PR #327 でリポジトリ ID 生成が `<name>_` になる不具合として発現
  - PreToolUse は Read/Write/Bash 全呼び出しで発火するため、early exit より前に `git rev-parse + shasum` を置くと無関係なツール呼び出しで毎回外部プロセスが 2 つ走る
  - `vibecorp_stamp_path` が内部で `vibecorp_stamp_dir` を呼ぶ設計の場合、複数回呼ぶと同回数 `git rev-parse + shasum` が実行される。STAMP_DIR を一度取得して再利用することで O(N) → O(1) になる
  - `(allow file-write* (subpath "~/.cache/vibecorp"))` は存在しない中間ディレクトリの作成権限を含まない。macOS sandbox-exec の `subpath` は対象パスが存在するディレクトリの配下のみをカバーする
- **代替案**: `basename` の newline 除去に `tr -d '\n'` を `basename` と `tr -cs` の間に挟む方法も機能するが、`printf '%s'` の方がコマンド置換の標準的な newline 除去慣用句として可読性が高い

## 2026-04-17 — 原子的置換パターンの実挙動確認とSBPL設計（Issue #329）

## 2026-04-18 — CodeRabbit による cross-PR 統合問題検出の技術評価

## 2026-04-29 — Source of Truth 起点の値同期検証パターン

`tests/test_diagnose_limits_sync.sh`（Issue #247）で導入した、`vibecorp.yml` を Source of Truth として5ファイル間（`vibecorp.yml`・`docs/cost-analysis.md`・`docs/specification.md`・`.claude/rules/autonomous-restrictions.md`・`tests/test_diagnose_limits.sh`）の値同期を検証するパターンを記録する。

- **パターン**: vibecorp.yml から基準値を読み取り、他ファイルの参照値と突き合わせる。ファイルごとに記述形式（YAML / Markdown テーブル / シェル条件式）が異なるため、それぞれ専用の抽出関数を用意する
- **awk による Markdown テーブル抽出（`extract_md_table_value`）**: 行頭 `|` でテーブル行に限定してマッチし、他のコンテキストへの誤ヒットを防ぐ

```bash
extract_md_table_value() {
  local file="$1" key="$2"
  awk -v key="$key" '
    /^\|/ && $0 ~ key {
      # | key | value | 形式から value を取り出す
      n = split($0, cols, "|")
      for (i = 2; i <= n; i++) {
        gsub(/^ +| +$/, "", cols[i])
        if (cols[i] ~ key) { gsub(/^ +| +$/, "", cols[i+1]); print cols[i+1]; exit }
      }
    }
  ' "$file"
}
```

- **YAML ブロック抽出**: `.claude/rules/shell.md` 準拠の `awk` ブロック単位抽出（次のトップレベルキーで停止）で `vibecorp.yml` から基準値を読む
- **技術的負債**: 5ファイルの参照箇所がバラバラな記述形式になっている点は解消が望ましいが、現状は sync テストで担保する方針
