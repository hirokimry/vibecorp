### 2026-03-22: settings.json マージ時のフック重複排除を install.sh に組み込む

- **判断**: install.sh の settings.json マージ処理に `unique_by(.command)` を追加し、フックエントリの重複を自動排除する
- **根拠**: セルフホスティング環境で install.sh を複数回実行した際に、同一フックが重複登録される問題が実際に発生した。ユーザーが手動で settings.json を編集して重複を取り除く必要があるのは UX として不適切であり、インストーラーが冪等性を保証すべき
- **代替案**: 実行前に既存設定の有無を確認してスキップする方式も検討したが、バージョン更新時にフック内容が変わる場合に対応できないため、マージ時の重複排除が適切と判断

### 2026-03-22: .claude/.gitignore の自動生成を install.sh が担う設計

- **判断**: install.sh に `generate_claude_gitignore` 関数を追加し、.claude/.gitignore を自動生成する。除外エントリは `plans/` のみ。既存ファイルがある場合は不足分のみ追記し、ユーザー独自エントリは保持する
- **根拠**: `plans/` 配下は一時的な実装計画ファイルであり、git 追跡すべきでない。しかしユーザーが手動で .gitignore を作成・管理するのは導入摩擦になる。install.sh がセットアップを一括担う設計思想に沿って、.gitignore 生成もインストーラーの責務とした。既存ファイルを上書きしない設計により、ユーザーがカスタマイズした内容を壊さない
- **代替案**: CLAUDE.md にユーザーへの手順案内を記載するだけの方式も検討したが、指示を読まずに進める開発者が多いため、自動化が確実

### 2026-04-02: review-gate.sh の README 未反映を要更新と判定

- **判断**: Issue #209 で追加された `review-gate.sh` は README のフック一覧・スタンプ対応表・settings.json 掲載内容の3箇所に反映が必要と判定した
- **根拠**: README は「インストールされるもの」の Source of Truth として機能しており、フック一覧に掲載されていないフックはユーザーが存在を認識できない。透明性のバリュー（「全ての判断根拠をルール・設定ファイルとして可視化する。ブラックボックスを作らない」）に反する状態である
- **代替案**: 新フック追加時に README 更新を省略する方式も考えられるが、README がフックの公式一覧として機能している以上、省略は情報の欠落に当たる
