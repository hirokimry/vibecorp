### 2026-03-22: settings.json マージ時のフック重複排除を install.sh に組み込む

- **判断**: install.sh の settings.json マージ処理に `unique_by(.command)` を追加し、フックエントリの重複を自動排除する
- **根拠**: セルフホスティング環境で install.sh を複数回実行した際に、同一フックが重複登録される問題が実際に発生した。ユーザーが手動で settings.json を編集して重複を取り除く必要があるのは UX として不適切であり、インストーラーが冪等性を保証すべき
- **代替案**: 実行前に既存設定の有無を確認してスキップする方式も検討したが、バージョン更新時にフック内容が変わる場合に対応できないため、マージ時の重複排除が適切と判断

### 2026-03-22: .claude/.gitignore の自動生成を install.sh が担う設計

- **判断**: install.sh に `generate_claude_gitignore` 関数を追加し、.claude/.gitignore を自動生成する。除外エントリは `plans/` のみ。既存ファイルがある場合は不足分のみ追記し、ユーザー独自エントリは保持する
- **根拠**: `plans/` 配下は一時的な実装計画ファイルであり、git 追跡すべきでない。しかしユーザーが手動で .gitignore を作成・管理するのは導入摩擦になる。install.sh がセットアップを一括担う設計思想に沿って、.gitignore 生成もインストーラーの責務とした。既存ファイルを上書きしない設計により、ユーザーがカスタマイズした内容を壊さない
- **代替案**: CLAUDE.md にユーザーへの手順案内を記載するだけの方式も検討したが、指示を読まずに進める開発者が多いため、自動化が確実

### 2026-04-02: protect-branch.sh を全プリセット（minimal 以上）に追加

- **判断**: `protect-branch.sh` をファイル保護型フックとして全プリセットに配置し、README.md のフック一覧・プリセットテーブル・settings.json サンプルの3箇所に反映した
- **根拠**: メインブランチへの直接 Edit/Write/git commit は「規律の自動化」バリューに反する。`protect-files.sh` と同カテゴリ（ファイル保護型）として minimal 以上で有効化するのが一貫性のある判断。プリセットテーブルでは `protect-files` の直後に並べ、settings.json サンプルでも同マッチャー内の隣接エントリに配置した
- **代替案**: standard 以上での提供も検討したが、ブランチ保護はチーム規模に依らず基本的な安全網であり、個人開発でも有効なため minimal 以上が適切

### 2026-04-02: review-gate.sh の README 未反映を要更新と判定

- **判断**: Issue #209 で追加された `review-gate.sh` は README のフック一覧・スタンプ対応表・settings.json 掲載内容の3箇所に反映が必要と判定した
- **根拠**: README は「インストールされるもの」の Source of Truth として機能しており、フック一覧に掲載されていないフックはユーザーが存在を認識できない。透明性のバリュー（「全ての判断根拠をルール・設定ファイルとして可視化する。ブラックボックスを作らない」）に反する状態である
- **代替案**: 新フック追加時に README 更新を省略する方式も考えられるが、README がフックの公式一覧として機能している以上、省略は情報の欠落に当たる

### 2026-04-02: command-log.sh をゲートフックではなくログ専用フックとして実装

- **判断**: `command-log.sh` は実行の可否を判定するゲートフック（ブロック型）ではなく、全コマンドをログに記録するだけの観察専用 PreToolUse フックとして実装した。ブロック判定は行わず、終了コード 0 で常に通過させる
- **根拠**: ゲートフックとして実装した場合、ユーザーが想定外のコマンドを実行できなくなる。これは「導入の手軽さ」バリュー（「設定の迷いをなくす」）に反し、初期導入の摩擦を高める。まず観察フェーズとしてログを蓄積し、`/approve-audit` スキルで棚卸しした上で allow リストに追加するという「観察→承認」の二段階フローが、「導入の手軽さ」と「規律の自動化」の両立を実現する
- **代替案**: 初回から allow リスト外コマンドをブロックする方式も検討したが、チームによって許可コマンドのセットが大きく異なるため、一律にブロックすると導入直後に大量の false positive が発生し UX が損なわれる。段階的に規律を育てる設計のほうが「段階的成長」バリューにも合致する

### 2026-04-11: /spike-loop を full プリセット専用スキルとして承認

- **判断**: `templates/claude/skills/spike-loop/SKILL.md` で追加された `/spike-loop` スキルを full プリセット専用スキルとして承認し、README.md のスキル一覧テーブルと「主要スキルの使い方」セクションに反映した
- **根拠**: `/spike-loop` は `/ship-parallel` の E2E 検証に COO エージェント（TeamCreate での並列実行）を必要とするため、full プリセット専用の位置付けは妥当。修正の自動適用を行わず分析レポートのみ出力する Phase 1 設計は、「暴走を防止する」という既存の `/diagnose` と同じ原則（起票と実装の分離）に沿っており、MVV の「規律の自動化」バリューと整合する。README への追記は「透明性」バリュー（「全ての判断根拠をルール・設定ファイルとして可視化する」）の観点から必須
- **代替案**: Phase 2（修正の自動適用）まで README 掲載を保留する案も考えられるが、スキル自体は既に追加済みであり、ユーザーが存在を認識できない状態は情報の欠落に当たるため、Phase 1 の段階で反映することが適切

### 2026-04-11: コンテナ隔離環境を vibecorp パッケージの一部として提供する方針を承認

- **判断**: CEO 方針「`--dangerously-skip-permissions` を前提としたコンテナ隔離環境を vibecorp パッケージの一部として提供する」をプロダクト観点で承認する。ただし「導入の手軽さ」バリューとの両立のため、コンテナ前提は full プリセット限定とし、Phase 2（テンプレート化）のタイミングで install.sh に組み込む。
- **根拠**: ミッション「AIエージェントを組織化し再現性のあるプロダクト開発を実現する」および「完全自律でプロダクトを回す」長期ビジョンに整合する。コンテナ隔離は「規律の自動化」バリューの延長線上にある機械的な規律。「段階的成長」バリューにより minimal/standard はコンテナ不要を維持し、full プリセット選択時のみ適用することで「ワンコマンド完結」との両立が可能。
- **代替案**: 全プリセットにコンテナを強制する案は「導入の手軽さ」バリューに反するため却下。Phase 1 でテンプレート化する案は知見蓄積前であり「段階的成長」バリューに反するため Phase 2 に先送り。
- **ロードマップ**: Phase 1（現行）= spike-loop 限定・コンテナ任意。Phase 2 = install.sh に Dockerfile/docker-compose.yml を組み込み・full プリセット時に自動配置。Phase 3 = 全ヘッドレス+対話のコンテナ統合。
- **設計上の未解決事項**: ship-parallel の worktree ↔ コンテナ間のマウント設計。Phase 2 開始前に CTO と合意が必要。
